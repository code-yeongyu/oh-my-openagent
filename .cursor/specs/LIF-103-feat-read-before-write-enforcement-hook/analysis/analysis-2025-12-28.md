## 🔴 CRITICAL Issues

### CRIT-1: Write-Then-Edit Workflow Logic Bug

**Location**: spec.md (US-4, FR-3, DD-3), plan.md (enforcement flow)

**Problem**: The spec states in US-4:
> "Editing a file immediately after creating it works without explicit read"

But the current enforcement logic **does NOT support this**:

1. Agent creates a new file with `write` tool → Allowed (file doesn't exist)
2. File now exists on disk
3. Agent immediately edits with `edit` tool → **BLOCKED**
   - Enforcement checks: Does file exist? **YES** (it was just created)
   - Was the file read? **NO** (only `read` tool tracks to registry)
   - Result: **Blocked despite the agent just creating it**

**Why This Matters**: This is a common workflow pattern. Creating a file template then refining it with edits should work seamlessly.

**Recommended Fix**: Track successful `write` operations as if the file was "read". When a file is written, the agent clearly has full context about its content. Add this to the enforcement logic:

```typescript
// In tool.execute.before, after tracking reads:
if (toolLower === "write") {
  const filePath = output.args.filePath ?? output.args.file_path;
  if (filePath && typeof filePath === "string") {
    // If this is creating a new file (doesn't exist), track it as "read" 
    // so subsequent edits are allowed
    const absolutePath = path.resolve(filePath);
    if (!existsSync(absolutePath)) {
      // File will be created - track as read for subsequent edits
      registry.recordRead(input.sessionID, absolutePath);
    }
  }
}
```

**Alternative**: Track writes to a separate "written files" set and check both during enforcement.

### MED-2: Synchronous File Existence Check

**Location**: plan.md (enforcement flow), spec.md (DD-3)

**Problem**: Using `existsSync()` is a synchronous blocking operation. While acceptable for the stated use case, it could add latency in environments with slow filesystems.

**Recommendation**: Consider using async `fs.access()` wrapped in try/catch, which is more consistent with the async hook pattern:

```typescript
async function fileExists(path: string): Promise<boolean> {
  try {
    await fs.promises.access(path);
    return true;
  } catch {
    return false;
  }
}
```

**Impact**: Low - Current sync approach is acceptable given low frequency of writes.

### MED-3: LRU Eviction Doesn't Preserve Write Tracking

**Location**: spec.md (DD-1), plan.md (T1.3)

**Problem**: If the LRU cache evicts an old entry to make room, that file will no longer be considered "read". A subsequent edit to that evicted file would be blocked even though the agent had previously read it.

**Scenario**:
1. Agent reads file A (earliest entry)
2. Agent reads 9,999 more files
3. Agent reads file B (triggers LRU eviction of file A)
4. Agent tries to edit file A → **Blocked** (A was evicted from cache)

**Recommendation**: 
- This is acceptable for the 10,000 file limit (very rare edge case)
- Document this limitation in Known Limitations section
- Consider logging when eviction happens: `log("[read-before-write] LRU eviction: ${path}")`

**Location**: plan.md (Phase 2 enforcement), spec.md (FR-5)

**Observation**: The spec mentions using `minimatch` for path exemption patterns. For performance, compiled patterns should be cached rather than recompiling on every enforcement check.

**Recommendation**: Cache compiled minimatch instances:

```typescript
const compiledPatterns = config.exempt_paths.map(p => new Minimatch(p));
// Then reuse in isPathExempt()
```

### LOW-2: Tool Name Case Sensitivity

**Location**: plan.md (T2.1 code), spec.md (FR-2)

**Observation**: The code does `toolLower = input.tool.toLowerCase()` for comparison but uses `config.exempt_tools.includes(input.tool)` for exemption check.

**Problem**: If config has `"LSP_RENAME"` but tool is `"lsp_rename"`, the exemption won't match.

**Recommendation**: Normalize exempt_tools at config load time or use case-insensitive comparison:
```typescript
const exemptToolsLower = config.exempt_tools.map(t => t.toLowerCase());
if (exemptToolsLower.includes(toolLower)) return;
```
