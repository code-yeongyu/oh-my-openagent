## 简介
本文件系统性梳理 Oh My OpenCode 中"实用工具钩子"的实现与工作机制，重点覆盖以下能力：
- 思维模式钩子：关键词触发、模型切换与思维配置注入、会话状态管理与清理
- 关键词检测器：多语言关键词匹配、模式识别、响应触发与上下文注入
- 非交互环境检测器：环境识别、交互命令拦截与非交互适配策略
- 交互式 Bash 会话钩子：tmux 会话管理、命令解析与状态跟踪、会话提醒与清理
- **LSP 诊断强制执行器**：文件修改跟踪、诊断验证、任务完成保护与错误处理
- **阶段流程强制执行器**：工作流阶段监控、过渡验证、状态恢复与异常处理

## 核心组件
- 思维模式钩子：在聊天参数阶段检测"think"类关键词，必要时将模型升级到高推理变体并注入对应提供商的思维配置；同时维护每会话的状态并在会话删除事件中清理。
- 关键词检测器：在消息阶段对用户输入进行关键词匹配，支持多语言；根据会话主从关系与关键词类型决定是否提升到"ultrawork"模式并注入上下文；可向上下文收集器注册关键词信息。
- 非交互环境检测器：在工具执行前拦截 bash 命令，识别交互式命令并给出警告；对 git 命令在非交互环境中自动前置非交互环境变量，避免阻塞。
- 交互式 Bash 会话钩子：解析 tmux 命令，跟踪 omo- 前缀会话的创建/销毁/服务器关闭；持久化状态并在工具输出后追加会话提醒；在会话删除事件中清理所有跟踪会话。
- **LSP 诊断强制执行器**：跟踪文件修改状态，确保在任务完成前运行 LSP 诊断；当检测到未运行诊断的任务完成时，自动注入提醒消息；具备健壮的错误处理机制。
- **阶段流程强制执行器**：监控 boulder.json 状态变更，验证工作流阶段的正确过渡顺序；支持从现有状态恢复，提供详细的错误信息和修复指导。

## 详细组件分析

### 思维模式钩子（Think Mode）
- 触发条件：在聊天参数阶段提取文本，去除代码块后检测"think/ultrathink"等关键词。
- 切换机制：
  - 若当前模型已是高推理变体则跳过。
  - 否则查找高推理变体映射并替换模型引用。
  - 根据提供商与模型能力注入对应的思维配置（如 Anthropic 的 thinking、Google 的 thinkingLevel、OpenAI 的 reasoning_effort 等）。
- 状态管理：以 Map 维护每会话的请求标记、模型是否已切换、是否已注入思维配置；在会话删除事件中清理。
- 复杂度：检测与映射均为线性扫描，整体 O(n)；Map 访问为均摊 O(1)。

```mermaid
flowchart TD
Start(["进入 chat.params"]) --> Extract["提取文本并去代码块"]
Extract --> HasKeyword{"是否包含 think 关键词?"}
HasKeyword --> |否| SaveState["保存初始状态并返回"]
HasKeyword --> |是| CheckModel["读取当前模型(providerID/modelID)"]
CheckModel --> IsHigh{"是否已是高推理变体?"}
IsHigh --> |是| SaveState
IsHigh --> |否| GetVariant["查找高推理变体"]
GetVariant --> HasVariant{"存在高推理变体?"}
HasVariant --> |否| InjectCfg["尝试注入思维配置"]
HasVariant --> |是| SwitchModel["替换模型为高推理变体"]
SwitchModel --> InjectCfg["注入思维配置"]
InjectCfg --> SaveState["更新并保存会话状态"]
SaveState --> End(["结束"])
```

### 关键词检测器（Keyword Detector）
- 匹配策略：对文本去除代码块后，使用一组正则表达式匹配多语言关键词（ultrawork/search/analyze/brainstorm/consult-metis），支持动态消息生成。
- 响应触发：
  - 当检测到 ultrawork 且当前会话为主会话或非主会话仅保留 ultrawork 时，提升到最大精度（message.variant=max），并通过 TUI 弹出成功提示。
  - 可选地通过上下文收集器注册关键词，用于后续注入或统计。
- 会话过滤：背景任务会话（subagent）跳过关键词注入，避免误触发 Prometheus 限制；非主会话仅允许 ultrawork 类关键词生效。

```mermaid
flowchart TD
Enter(["进入 chat.message"]) --> Extract["提取文本并去代码块"]
Extract --> IsSys{"是否系统指令?"}
IsSys --> |是| Skip["跳过处理"]
IsSys --> |否| Detect["检测关键词(含类型)"]
Detect --> Any{"是否检测到关键词?"}
Any --> |否| End
Any --> |是| BGCheck{"是否背景任务会话?"}
BGCheck --> |是| End
BGCheck --> |否| MainCheck{"是否非主会话?"}
MainCheck --> |是| Filter["仅保留 ultrawork 类型"]
Filter --> Empty{"是否为空?"}
Empty --> |是| End
Empty --> |否| Activate["激活 Ultrawork 模式"]
MainCheck --> |否| Activate
Activate --> Toast["TUI 提示"]
Toast --> Collect["可选：注册上下文"]
Collect --> Log["记录日志"]
Log --> End
```

### 非交互环境检测器（Non-Interactive Env）
- 环境识别：通过环境变量与 TTY 判断是否处于 CI/GitHub Actions/非交互运行等场景。
- 命令拦截：拦截 bash 工具调用，若命令属于交互式（如编辑器、分页器、REPL、交互式 git 模式），则提示可能挂起。
- 适配策略：对 git 命令在非交互环境下自动前置一组非交互环境变量（如禁用编辑器、分页器、交互式提示等），避免阻塞。

```mermaid

IsGit --> |否| End
IsGit --> |是| NonInt{"是否非交互环境?"}

NonInt --> |是| Prefix["构建非交互环境前缀"]

Prepend --> Log["记录日志"]
```

### 交互式 Bash 会话钩子（Interactive Bash Session）
- 命令解析：对 tmux 命令进行分词与子命令识别，支持全局选项跳过与标志位提取（-s/-t 等），并规范化会话名（去除窗口/面板后缀）。
- 会话管理：跟踪 omo- 前缀会话集合，支持新建、销毁与服务器关闭三种操作；变更后持久化状态并更新时间戳。
- 状态跟踪：在工具执行后追加"活动会话"提醒；在会话删除事件中清理所有跟踪会话并清除本地持久化文件。
- 存储：以会话 ID 为文件名的 JSON 文件存储，字段包含会话集合与更新时间。

### LSP 诊断强制执行器（LSP Diagnostics Enforcer）
- **功能概述**：跟踪文件修改状态，确保在任务完成前运行 LSP 诊断，防止提交包含编译错误的代码。
- **文件修改跟踪**：监听 Edit、Write、MultiEdit 等工具的执行，自动记录修改的文件路径，并标记需要运行诊断。
- **诊断状态管理**：当检测到 lsp_diagnostics 工具执行时，解析输出结果判断诊断是否通过，清理由此产生的文件修改记录。
- **任务完成保护**：当 todowrite 工具标记任务为 completed 时，如果存在未验证的文件修改，则自动注入提醒消息，要求先运行诊断。
- **错误处理增强**：具备健壮的异常处理机制，包括 JSON 序列化容错、正则匹配容错、日志记录等，确保在各种异常情况下都能正常工作。

```mermaid
flowchart TD
Start(["tool.execute.after"]) --> TrackMod{"检测修改工具?"}
TrackMod --> |是| AddFile["添加到修改文件集合"]
AddFile --> ClearFlag["清除诊断通过标记"]
TrackMod --> |否| CheckDiag{"是否 lsp_diagnostics?"}
CheckDiag --> |是| ParseOutput["解析诊断输出"]
ParseOutput --> HasErrors{"检测到错误?"}
HasErrors --> |是| SetFailed["标记诊断失败"]
HasErrors --> |否| ClearFiles["清空修改文件集合"]
CheckDiag --> |否| CheckComplete{"是否 todowrite?"}
CheckComplete --> |是| HasCompleted{"是否有已完成任务?"}
HasCompleted --> |是| HasMods{"存在未验证修改?"}
HasMods --> |是| InjectMsg["注入诊断提醒"]
HasMods --> |否| End
HasCompleted --> |否| End
CheckComplete --> |否| End
End --> End(["结束"])
```

### 阶段流程强制执行器（Phase Flow Enforcer）
- **功能概述**：监控 Sisyphus 工作流的阶段状态，确保按照正确的顺序进行阶段转换，防止跳过关键步骤。
- **阶段状态监控**：只关注 Write 和 Edit 工具对 .sisyphus/boulder.json 的修改，从中提取当前阶段信息。
- **阶段过渡验证**：基于预定义的阶段顺序（idle → planning → reviewing → executing → awaiting_user → completed），验证每次阶段转换的有效性。
- **状态恢复机制**：在会话启动时尝试从现有 boulder 状态恢复最后已知阶段，确保连续性。
- **异常处理增强**：具备完整的 try-catch 机制，包括 JSON 解析容错、正则匹配容错、状态恢复失败的降级处理等。

```mermaid

CheckTools --> |否| End
CheckTools --> |是| CheckFile{"是否 boulder.json?"}
CheckFile --> |否| End
CheckFile --> |是| ParseResult["解析输出结果"]
ParseResult --> ExtractPhase["提取阶段信息"]
ExtractPhase --> Validate{"验证阶段转换"}
Validate --> |有效| UpdateState["更新最后已知阶段"]
Validate --> |无效| InjectWarning["注入阶段跳过警告"]
UpdateState --> End
InjectWarning --> End
End --> End(["结束"])
```

## 性能考量
- 正则匹配与数组扫描：关键词与 think 关键词检测均为线性扫描，复杂度 O(n)；正则数量有限，实际开销较小。
- 字符串预处理：去代码块与文本拼接操作为线性，对长文本需注意内存占用。
- 模型映射与提供商解析：哈希表查找与集合判定为均摊 O(1)，性能稳定。
- I/O 操作：交互式 Bash 会话状态读写为磁盘 I/O，建议控制写频率（变更时才写）。
- **LSP 诊断钩子**：使用 Set 进行文件路径去重，Map 进行会话状态管理，内存占用与修改文件数量成正比。
- **阶段流程钩子**：仅在 Write/Edit 工具修改 boulder.json 时触发，JSON 解析和正则匹配开销很小。

## 故障排查指南
- 思维模式未生效
  - 检查输入是否包含 think/ultrathink 关键词且未被代码块包裹。
  - 确认当前模型是否已是高推理变体或提供商是否在支持列表中。
  - 查看日志中模型切换与配置注入记录。
- 关键词未触发
  - 确认消息非系统指令；检查是否在背景任务会话中；非主会话仅保留 ultrawork 生效。
  - 核对关键词正则是否覆盖目标语言与形态。
- 非交互环境仍出现交互等待
  - 检查环境变量是否正确设置；确认命令是否命中禁止交互命令。
  - 对 git 命令确保前置了非交互环境变量。
- 交互式 Bash 会话残留
  - 检查工具输出是否包含错误前缀；确认会话名称是否以 omo- 开头。
  - 在会话删除事件后检查持久化文件是否清理。
- **LSP 诊断强制执行器问题**
  - 检查是否正确监听了 Edit、Write、MultiEdit 工具的执行。
  - 确认 lsp_diagnostics 输出格式是否符合预期的错误检测规则。
  - 验证 todowrite 是否正确传递了任务状态信息。
  - 查看日志中关于诊断状态更新的记录。
- **阶段流程强制执行器问题**
  - 确认 boulder.json 路径是否正确（包含 .sisyphus/ 前缀）。
  - 检查 Write/Edit 工具的输出格式是否包含 phase 字段。
  - 验证阶段顺序常量定义是否完整。
  - 查看会话启动时的状态恢复是否成功。

## 附录：配置与使用场景
- 思维模式钩子
  - 触发条件：用户输入包含 think/ultrathink 等关键词。
  - 适用场景：需要更强推理能力的任务（如复杂分析、架构设计）。
  - 关键配置点：提供商别名解析、模型 ID 归一化、高推理变体映射、思维配置注入。
- 关键词检测器
  - 触发条件：检测到 ultrawork/search/analyze/brainstorm/consult-metis 等关键词。
  - 适用场景：需要特定工作流引导的对话（如最大化代理利用、并行搜索、深度分析、设计先于实现）。
  - 关键配置点：关键词正则与消息模板、动态消息生成函数、会话过滤规则。
- 非交互环境检测器
  - 触发条件：在非交互环境中执行 bash 工具。

  - 触发条件：执行 interactive_bash 工具并传入 tmux 命令。
  - 适用场景：本地 tmux 会话管理与提醒、跨进程会话追踪。

  - 适用场景：确保代码质量，防止提交包含编译错误的代码。

- **阶段流程强制执行器**
  - 触发条件：Write/Edit 工具修改 .sisyphus/boulder.json 文件。
  - 适用场景：强制遵循 Sisyphus 工作流的正确阶段顺序。
  - 关键配置点：阶段顺序定义、有效转换规则、警告消息模板、状态恢复机制。
