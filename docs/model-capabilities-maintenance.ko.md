# 모델 기능(Capabilities) 유지 관리

이 프로젝트는 모델 기능(capability) 해석을 계층화된 시스템으로 다룹니다.

1. 연결된 프로바이더의 런타임 메타데이터
2. `models.dev`의 번들/런타임 스냅샷 데이터
3. 명시적 호환성 별칭(alias)
4. 최후의 수단인 휴리스틱 폴백

## 내부 정책

- 빌트인 OmO 에이전트/카테고리 요구 모델은 정규(canonical) 모델 ID를 사용해야 합니다.
- 별칭은 과거 OmO 이름이나 프로바이더별 데코레이션과의 호환성을 유지하기 위해서만 존재합니다.
- 정규 모델 ID와 구조화된 설정으로 동일한 것을 표현할 수 있다면, `-high`, `-low`, `-thinking` 같은 데코레이션이 붙은 새 이름을 빌트인 요구 사항에 추가해서는 안 됩니다.
- 프로바이더나 설정 입력이 여전히 별칭을 사용한다면 가장자리(edge)에서 정규화하고 내부적으로는 정규 ID로 계속 진행합니다.

## 별칭 추가 시

- `src/shared/model-capability-aliases.ts`에 별칭 규칙을 추가합니다.
- 별칭이 존재하는 이유를 함께 명시합니다.
- 별칭이 명시적으로 커버되도록 테스트를 추가하거나 갱신합니다.
- 별칭의 정규 타깃이 번들된 `models.dev` 스냅샷에 존재하는지 확인합니다.

## 가드레일

`bun run test:model-capabilities`는 다음 불변식(invariants)을 강제합니다.

- exact 별칭 타깃은 번들된 스냅샷에 반드시 존재해야 함
- exact 별칭 키는 정규 `models.dev` ID로 조용히 변환되면 안 됨
- 패턴 별칭은 정규 스냅샷 ID를 재작성해서는 안 됨
- 빌트인 요구 모델은 정규 상태를 유지하고 스냅샷에 의해 뒷받침되어야 함

스케줄링된 `refresh-model-capabilities` 워크플로우는 자동 스냅샷 리프레시 PR을 열기 전에 이 가드레일들을 실행합니다.
