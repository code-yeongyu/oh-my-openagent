# 매니페스토

Oh My OpenAgent의 원칙과 철학.

---

## 인간의 개입은 실패 신호다

**HUMAN IN THE LOOP = 병목**

자율 주행을 생각해보세요. 사람이 운전대를 잡아야 한다면 그것은 기능이 아니라 시스템의 실패입니다. 차가 그 상황을 스스로 처리하지 못한 것입니다.

**왜 코딩이라고 다를까요?**

당신이 다음과 같은 상황에 있다면:
- AI가 반쯤 만들다 만 코드를 고치고 있음
- 명백한 실수를 수동으로 교정하고 있음
- 작업 단계마다 에이전트를 직접 안내하고 있음
- 동일한 요구 사항을 반복적으로 명확화하고 있음

그것은 "인간-AI 협업"이 아닙니다. AI가 자기 일을 못 하고 있는 것입니다.

**Oh My OpenAgent는 이 전제 위에 만들어졌습니다**: 에이전트 작업 중 인간의 개입은 본질적으로 잘못된 신호입니다. 시스템이 올바르게 설계되었다면, 에이전트는 당신이 보모처럼 지켜보지 않아도 작업을 완수해야 합니다.

---

## 구별 불가능한 코드

**목표: 에이전트가 작성한 코드는 시니어 엔지니어가 작성한 코드와 구별 불가능해야 한다.**

"정리가 필요한 AI 생성 코드"가 아닙니다. "괜찮은 출발점"이 아닙니다. 실제로, 최종적으로, 프로덕션 준비가 된 코드입니다.

이것이 의미하는 바:
- 기존 코드베이스 패턴을 정확히 따름
- 요청 없이도 적절한 에러 처리
- 실제로 올바른 것을 테스트하는 테스트
- AI slop 없음 (과잉 엔지니어링, 불필요한 추상화, 스코프 크리프)
- 가치를 더할 때만 주석

커밋이 사람이 한 것인지 에이전트가 한 것인지 구분할 수 있다면, 에이전트는 실패한 것입니다.

---

## 토큰 비용 vs 생산성

**생산성을 크게 높일 수 있다면 더 높은 토큰 사용은 받아들일 만하다.**

다음을 위해 더 많은 토큰을 쓰는 것:
- 여러 전문 에이전트가 병렬로 리서치
- 인간 개입 없이 작업을 완전히 끝내기
- 완료 전에 작업을 철저히 검증
- 작업 간 지식 축적

10배, 20배, 100배의 생산성 향상을 의미한다면 그것은 가치 있는 투자입니다.

**하지만:**

불필요한 토큰 낭비를 추구하지 않습니다. 시스템은 다음을 최적화합니다.
- 단순 작업에 더 저렴한 모델(Haiku, Flash) 사용
- 중복 탐색 회피
- 세션 간 학습 캐시
- 충분한 컨텍스트가 모이면 리서치 중단

토큰 효율은 중요합니다. 그러나 작업 품질이나 인간의 인지 부담을 희생하면서까지는 아닙니다.

## 인간의 인지 부담을 최소화하라

**인간은 원하는 것만 말하면 된다. 나머지는 모두 에이전트의 일이다.**

이를 달성하는 두 가지 접근법이 있습니다.

### 접근법 1: Prometheus (인터뷰 모드)

당신이 말합니다: "인증을 추가하고 싶어."

Prometheus는:
- 코드베이스를 조사해 기존 패턴을 이해
- 실제 발견 사항을 기반으로 명확화 질문
- 당신이 고려하지 못한 엣지 케이스를 표면화
- 결정을 내릴 때마다 문서화
- 완전한 작업 계획 생성

**당신은 의도를 제공한다. 에이전트는 구조를 제공한다.**

### 접근법 2: Ultrawork (그냥 해줘 모드)

당신이 말합니다: "ulw add authentication"

에이전트는:
- 올바른 접근 방식을 파악
- 베스트 프랙티스 리서치
- 컨벤션을 따라 구현
- 모든 것이 작동하는지 검증
- 완료될 때까지 계속 진행

**당신은 의도를 제공한다. 에이전트는 모든 것을 처리한다.**

두 경우 모두, 인간의 일은 **원하는 것을 표현하는 것**이지 어떻게 진행될지 관리하는 것이 아닙니다.

## 예측 가능, 연속적, 위임 가능

**이상적인 에이전트는 컴파일러처럼 동작해야 한다**: 마크다운 문서가 들어가면 동작하는 코드가 나온다.

### 예측 가능

같은 입력이 주어지면:
- 동일한 코드베이스 패턴
- 동일한 요구 사항
- 동일한 제약

출력은 일관되어야 합니다. 무작위가 아니고, 놀라운 것이 아니며, 요청하지 않은 방식의 "창의성"이 아닙니다.

### 연속적

작업은 중단을 견뎌내야 합니다.
- 세션이 죽었다? `/start-work`로 재개
- 자리를 비워야 한다? 진행 상황이 추적됨
- 며칠짜리 프로젝트? 컨텍스트가 보존됨

에이전트가 상태를 유지합니다. 당신이 할 필요가 없습니다.

### 위임 가능

유능한 팀원에게 작업을 맡기고 처리를 신뢰하듯이, 에이전트에게도 위임할 수 있어야 합니다.

이것이 의미하는 바:
- 명확한 수용 기준, 독립적으로 검증
- 무언가 잘못되면 자체 수정 동작
- 정말로 필요할 때만 (Oracle, 사용자에게) 에스컬레이션
- "거의 완료"가 아닌 완전한 작업

## 핵심 루프

```
Human Intent → Agent Execution → Verified Result
       ↑                              ↓
       └──────── Minimum ─────────────┘
          (intervention only on true failure)
```

Oh My OpenAgent의 모든 것은 이 루프가 동작하도록 설계되었습니다.

| 기능 | 목적 |
|---------|---------|
| Prometheus | 지능적 인터뷰로 의도 추출 |
| Metis | 모호함이 버그가 되기 전에 포착 |
| Momus | 실행 전 계획이 완전한지 검증 |
| Orchestrator | 인간의 마이크로매니지먼트 없이 작업 조율 |
| Todo Continuation | 완료를 강제, "다 했어요" 거짓말 방지 |
| Category System | 인간 결정 없이 최적 모델로 라우팅 |
| Background Agents | 사용자를 막지 않고 병렬 리서치 |
| Wisdom Accumulation | 작업으로부터 배움, 실수를 반복하지 않음 |

---

## 실제로 이것이 의미하는 바

**당신은 다음을 할 수 있어야 합니다:**

1. 원하는 것을 설명한다 (높은 수준이든 상세하든, 당신의 선택)
2. 필요하면 에이전트가 인터뷰하도록 둔다
3. 계획을 확정한다 (또는 그냥 ultrawork에게 맡긴다)
4. 자리를 뜬다
5. 완료되고 검증된, 프로덕션 준비가 된 작업을 받아든다

**이것이 안 된다면, 시스템의 무언가가 개선되어야 합니다.**

## 우리가 만들고 있는 미래

다음과 같은 세상:
- 인간 개발자는 **무엇을** 만들지에 집중하고, AI에게 만들도록 시키는 **방법**에 집중하지 않음
- 코드 품질이 누가(또는 무엇이) 작성했는지와 무관함
- 복잡한 프로젝트가 단순한 프로젝트만큼이나 쉬움 (시간만 더 걸릴 뿐)
- "프롬프트 엔지니어링"이 "컴파일러 디버깅"만큼이나 사라짐

**에이전트는 보이지 않아야 합니다.** 숨겨져 있다는 의미가 아니라, 그냥 동작한다는 의미에서요. 전기처럼, 수도처럼, 인터넷처럼.

스위치를 누른다. 불이 켜진다. 전력망에 대해 생각하지 않는다.

그것이 목표입니다.
